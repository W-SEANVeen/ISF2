## 一、全局阻尼 —— 战斗核心循环

### 推荐位置
替换当前第10页"核心玩法：玩家射击、近身搏斗"（该页内容与游戏实际机制不符）

### 标题
**核心玩法：全局阻尼 —— 压制与反压制**

### 核心描述

这是整款游戏**最核心的玩法闭环**。在《孤城》中，每一次击杀不是局部事件——它会影响整个战场。这个机制叫做**"全局阻尼"**：一个 0.1 ~ 1.0 之间浮动的全局系数，直接控制所有敌方单位的推进速度。

> 设计意图：**将玩家的射击精度转化为生存时间的可控变量**——射得越准，战场压力就越低；射不准或不敢射，敌人就越逼越近。这不是"隐藏机制"，而是玩家能直观感受到的因果关系。

### 核心循环图（建议页面中央大图）

```
     玩家射中敌军
         │
         ▼
  globalDamping = 0.1  ← 瞬间触发
         │
         ▼
  全军速度 × 0.1，几乎停滞  ← 深呼吸窗口
         │
         ▼
  阻尼以 0.5/s 缓慢恢复    ← 大约 2 秒有效窗口
         │
         ▼
  压力逐渐回升             ← 需要再次射杀
         │
         └─────────────→ 回到起点
```

### 被阻尼影响的三大系统

| 系统 | 公式 | 效果 |
|------|------|------|
| `SquadCommander` 步兵方阵 | `agent.speed = baseSpeed × damping` | 步兵停滞，方阵停止推进 |
| `EnemyHorse` 骑兵冲锋 | `chargeSpeed = baseSpeed × damping` | 骑兵减速，城门压力降低 |
| `SiegeManager` 攻城进度 | `progress += advanceRate × damping × Δt` | 梯子竖起速度变慢 |

### 关键代码

```csharp
public void TriggerSuppressingFire()
{
    globalDamping = 0.1f; // 一箭射出，全场冻结
}

void Update()
{
    // 所有推进速度 = 基础速度 × 阻尼系数
    siegeProgress += baseAdvanceSpeed * globalDamping * Time.deltaTime;
    
    // 阻尼随时间恢复（约 2 秒回到正常）
    if (globalDamping < 1f)
        globalDamping += dampingRecoveryRate * Time.deltaTime;
}
```

### Presenter 讲解节奏

1. "这不是一个射击游戏——这是一个**节奏管理游戏**。射得准 = 喘口气，射不准 = 被淹没"（定性）
2. "一个大黄弩命中，不只是打死一个敌人——阻尼瞬间降到 0.1，**全场敌人几乎停滞**"（展示因果链条）
3. "阻尼会按每秒 0.5 的速度恢复，所以大约 2 秒后压力又回来了——你必须再次射中"（讲循环）
4. "一行代码 `globalDamping = 0.1f` 同时控制了三个大系统——步兵走不了、骑兵冲不动、梯子架不起来"（技术亮点，四两拨千斤）

### 一句话总结

> "射中一箭，对面全停。射得越准，活得越久。"

## 二、音效系统 —— 战斗音与叙事音

### 推荐位置
替换当前第15页"技术实现：音效设计"

### 页面标题
**技术实现：两层级音效 —— 战斗音效与叙事音效**

### 核心描述

整套音效系统由 AudioManager 统一调度，底层采用 8 通道挂载音效池实现空间音频。AudioManager 在场景启动时预生成 8 个 AudioSource 存入队列，需要播放音效时从队列取出一个，通过 PlayAttached 接口将音源挂载到目标物体的 Transform 上，随目标在空间中运动；播放完毕后由协程自动将 AudioSource 归还到队列，供下一次使用。所有音效文件通过 ScriptableObject 资源库以 key-value 方式集中管理，代码按 key 名查找调用，不需要关心具体音频文件路径。音源的空间定位统一采用 spatialBlend = 0.6 的混合参数——兼顾 3D 空间方位感与移动端的清晰度，避免纯 3D 在 VR 中声音发闷的问题。这套底层架构同时服务于两套音效：战斗音效和叙事音效。

### 卡片一：战斗音效

**标题**：战斗音效 —— 箭矢与弩箭的空间定位

**正文**：
敌方弓箭从拉弓、破空到命中，有完整的音效链条。弓弦绷紧的吱嘎声、箭矢离弦的破空呼啸、命中城墙或地面的撞击声，层层递进形成压迫感。这些音效通过 PlayAttached 机制挂载到每一支箭矢上——当箭矢从对象池取出发射时，AudioManager 从 8 通道池借出一个 AudioSource，将其父级设为箭矢的 Transform，音源即跟随箭矢在空间中飞行。玩家转头或移动时，能通过声音准确判断箭从哪个方向飞来。播放结束后，协程自动将 AudioSource 归还池中，即使箭矢在命中时被销毁，池对象也不会丢失。玩家射出的弩箭同样采用这一机制：弩箭射出时音源挂载在箭矢上，从弩口飞出到命中目标的全过程，声音始终跟随弹头位置，听觉与视觉同步锁定目标。每个 AudioSource 的 pitch 在 0.9~1.1 之间随机取值，避免大量箭矢同时发射时产生机械重复的听感。

### 卡片二：叙事音效

**标题**：叙事音效 —— 号角与战鼓的战场叙事

**正文**：
战斗开局时播放敌方进攻号角。两种不同调性的号角声交替响起，配合由远及近的音量衰减曲线，在听觉上勾勒出大军压境的空间层次。号角声属于一次性叙事音效，使用独立 AudioSource 一次性播放，不占用 8 通道池——因为这类音效不与任何运动物体绑定，不需要位置跟随。号角之后，城墙擂鼓声持续循环播放，鼓点的节奏与战况强度联动。联动逻辑通过代码在每帧更新中读取全局阻尼系数：阻尼越低（玩家射中敌人触发压制效果，敌军推进变慢），鼓声的播放速度随之降低，声音变得沉缓；阻尼恢复、敌军加速推进时，鼓声 pitch 逐渐拉高，节奏变得密集急促。玩家不需要看任何 UI 指示器，仅从听觉上就能感知当前战局是紧张还是缓和。

### Presenter 讲解节奏

1. "音频系统底层就干了三件事——8 个 AudioSource 循环用、声音挂到目标上跟着走、spatialBlend 调了个 0.6"（先一句话说清底层架构）
2. "箭雨 200 支箭共用 8 个 AudioSource 池——没有 GC 抖动，播完自动归还，箭被销毁了池对象也丢不了"（战斗音的亮点）
3. "号角用独立音源一次性播放不占池；鼓声跟全局阻尼联动，阻尼低鼓声缓，阻尼高鼓声急"（叙事音的联动设计）
4. "从头到尾不需要玩家操作任何开关——听觉反馈全自动融入战斗循环"（收束）

### 一句话总结

> "8 通道池兜住 200 支箭的空间定位音效，号角鼓声随战局自动变奏——听觉反馈就是 UI 指示器。"

## 三、天气系统 —— 下雪天气效果制作

### 推荐位置
替换当前第14页"视觉风格"（调整为下雪天气的制作流程）

### 页面标题
**天气系统：下雪天气效果制作 —— 粒子层 + Shader 积雪**

### 核心描述

下雪天气效果由空中飘雪、地面和城墙积雪三部分组成。空中飘雪采用两套 VFX Graph 粒子叠加实现——一套大雪片在场景不同位置放了两份铺满全场，另一套小雪粒子填隙。地面积雪通过 SG_Terrain 着色器控制，城墙积雪通过 mat_house 材质的 inkwall 着色器控制。三部分配合，覆盖从空中到地表的全部降雪视觉。

### 卡片一：两套粒子叠加

**标题**：空中飘雪：大小雪花叠加

**正文**：
空中飘雪不是单层粒子刷出来的，而是两套 VFX Graph 粒子叠加的效果。一套大雪片体积大、粒子数量多，负责覆盖战场的主要区域——在场景中不同位置放了两份，保证大雪的范围铺满全场。另一套小雪粒子少而密集，漂浮在大雪花之间的空隙里，让雪量看起来更饱满。两套粒子共用相同的发射区域、下落速度和噪声扰动，唯一区别是粒子大小和密度不同——大雪花铺量，小雪花填隙，叠加在一起形成大雪天里大小雪花交织飘落的视觉效果。

### 卡片二：地面 Shader 积雪

**标题**：地面 —— SG_Terrain 着色器

**正文**：
地面使用 SG_Terrain 着色器，通过 Snow Thickness 参数控制积雪厚度。雪面与裸露地面之间通过 Edge Noise 参数混合，边缘带有噪声扰动，避免积雪与非积雪区域的生硬界线。

### 卡片三：城墙 Shader 积雪

**标题**：城墙积雪：垂直面不均匀雪量覆盖

**正文**：
城墙积雪不是整个墙面均匀覆盖一层白，而是通过 inkwall 着色器的噪声扰动，在垂直面上形成**局部积雪、局部裸露**的不均匀效果。砖墙凸起处雪量少，露出砖墙原本的材质颜色；砖缝凹陷处和城墙顶部边缘雪量多，积雪堆积更厚，呈现白色。_Snow_Thickness 参数控制整体积雪程度——调高时积雪区域扩大，凹陷处最先变白，凸起处最后才被覆盖；调低时积雪收缩，只留在背风角落和砖缝里。配合 _snow 颜色参数调节积雪的白度与冷暖色调，以及噪声边缘混合，雪与非雪之间没有硬边界，而是带有破碎的不规则过渡，模拟真实积雪被风吹散、部分堆积在墙面凹陷处的自然状态，避免整面城墙被雪均匀覆盖的"塑料感"。

### Presenter 讲解节奏

1. "下雪天气分三部分——天上的雪用两套 VFX Graph 粒子叠加，地上的雪用地面 Shader，城墙也有一套独立的 Shader 积雪"（定性）
2. "大雪片在场景不同位置放了两份铺满全场，小雪花填在大雪花之间的空隙里，雪量才饱满"（两套粒子的分工）
3. "城墙积雪不是整个墙面刷白，砖缝凹陷处雪多、凸起处雪少，靠噪声控制不均匀分布——避免塑料感"（城墙积雪的视觉逻辑）
4. "地面也有单独的 SG_Terrain 参数控制积雪厚度，三部分合在一起才是完整的雪天场景"（收束）

### 一句话总结

> "两套 VFX Graph 粒子管天上飘的，两套 Shader 管地面和城墙积的——从空中到地表一套完整的雪天视觉方案。"

上半部分展示粒子叠加截图对比（大雪片/小雪粒子/叠加效果三张），中段展示城墙积雪局部特写（砖缝积雪 vs 砖面裸露），下半部分左右对比例图：左侧地面无积雪 vs 右侧地面有积雪（标注 Snow Thickness）。

## 四、NavMesh 寻路系统 —— Unity 寻路方案

### 推荐位置
新增独立技术页，放在第9页（敌军系统）之后

### 页面标题
**技术实现：Unity NavMesh 寻路系统**

### 核心描述

Unity NavMesh 的使用流程分为三步：烘培网格数据、挂载组件配置参数、代码调用 SetDestination 驱动寻路。

### 卡片一：烘培 NavMesh

**标题**：第一步 —— 生成寻路网格

**正文**：
场景中选中地面和可行走物体，设为 Navigation Static，调整参数后点击 Bake，生成蓝色覆盖的寻路网格。蓝色区域为单位可行走表面。

### 卡片二：NavMeshAgent 组件

**标题**：第二步 —— 挂载 Agent 并设置参数

**正文**：
在需要寻路的单位上添加 NavMeshAgent 组件，设置 Speed 和 Acceleration。多个单位同时移动时 Agent 自带避障功能。

### 卡片三：运行时寻路

**标题**：第三步 —— SetDestination 控制移动

**正文**：
调用 agent.SetDestination 传入目标坐标，Agent 自动寻路移动。通过 agent.speed 控制速度，agent.remainingDistance 判断到达。所有敌人的 speed 乘以全局阻尼系数，压制时全军减速。到达目的地后各自处理后续行为。

### 一句话总结

> "烘培网格 → 挂 Agent → SetDestination——三步走完寻路，三类敌人用法一致，差别只在到达后做什么。"

## 五、VR 接入流程 —— PICO Integration SDK 接入

### 推荐位置
新增独立技术页

### 页面标题
**技术实现：PICO Integration SDK 接入流程 —— 从注册到打包**

### 核心描述

PICO VR 设备接入 Unity 分为三个步骤：注册开发者账号并创建应用获取 APP ID、配置 Unity 开发环境并导入 SDK、完成项目参数设置。

### 卡片一：开发者账号与 APP ID

**标题**：第一步 —— 注册 PICO 开发者账号并创建应用

**正文**：
前往 PICO 开发者平台注册账号并获取 APP ID。APP ID 在 Unity 中通过菜单栏 PICO → Platform Settings 填入并应用。

### 卡片二：Unity 环境配置与 SDK 导入

**标题**：第二步 —— 配置 Unity 开发环境并导入 SDK

**正文**：
项目需使用 Unity 2022 LTS 或更高版本，Android SDK 最低 API Level 29。在 Unity 的 Package Manager 中点击 + 号选择 Add package from disk，选中 PICO Unity Integration SDK 文件夹中的 package.json 完成导入。导入成功后编辑器菜单栏会出现 PICO 选项卡，表示 SDK 挂载成功。

### 卡片三：项目参数设置

**标题**：第三步 —— XR 插件、Player 参数与 APP ID 配置

**正文**：
在 Project Settings 的 XR Plug-in Management 中，切换到安卓平台选项卡，勾选 PICO 作为 XR 插件，不要勾选其他平台插件。在 Player 设置中切换为安卓平台，将 Minimum API Level 设为 Android 10.0（API Level 29），Scripting Backend 选 IL2CPP，Target Architectures 只勾选 ARM64。最后通过 PICO → Platform Settings 填入第一步获取的 APP ID。包名建议按 com.公司名.产品名 的格式编写。
按照Project Validation的各项要求、建议逐个修改项目配置

### 卡片四：打包与真机部署

**标题**：第四步 —— 构建 APK 并安装到 PICO 设备

**正文**：
确保所有需要场景已添加到 Scenes In Build 列表。通过 USB 连接 PICO 设备，点击 Build And Run 生成 APK 文件并安装、运行。

### Presenter 讲解节奏

1. "第一步去 PICO 开发者平台注册拿 APP ID——没有这个 ID 包都装不进去"（账号配置）
2. "SDK 在 Package Manager 里导入，Project Settings 里勾上 PICO 的 XR 插件"（环境配置）
3. "Player 设置 IL2CPP + ARM64、API Level 29，填入 APP ID"（参数设置）
4. "Build APK 后用 ADB 装到真机——注意 SDK 版本和 Unity 版本要匹配"（打包部署）

### 一句话总结

> "注册账号拿 APP ID、导入 SDK、配置项目参数——三步走完 PICO 接入。"

## 六、VR 制作流程 —— XR 场景搭建

### 推荐位置
新增独立技术页，放在接入流程之后

### 页面标题
**技术实现：VR 场景制作流程 —— XR Origin · 射线交互 · UI 适配**

### 核心描述

SDK 接入完成后，在 Unity 中搭建 VR 交互场景需要完成三个步骤：创建 XR Origin 并配置手柄控制、挂载 XRRayInteractor 实现手柄射线、配置 Canvas 和 EventSystem 让 UI 支持手柄点击。

### 卡片一：搭建 XR 场景

**标题**：第一步 —— 创建 XR Origin 并配置手柄交互

**正文**：
导入 PICO SDK 和 XR Interaction Toolkit 后，在场景中创建 XR Origin（XR Rig）作为 VR 交互的根节点，替代 Unity 默认的 Main Camera。XR Origin 包含左右手控制器和摄像机，所有坐标以手柄追踪数据为输入。PICO 的 Building Blocks 菜单提供了 PICO Controller Tracking 预制体，可以直接生成已配置好的 XR Origin。之后通过 XR Interaction Toolkit 的 Starter Assets 导入默认输入动作绑定到控制器上。手柄模型可以选择 PICO SDK 提供的 LeftControllerModel 和 RightControllerModel 预制体替换默认模型。

### 卡片二：VR 射线交互

**标题**：第二步 —— 挂载 XRRayInteractor

**正文**：
在 XR Origin 的左右手控制器上挂载 XRRayInteractor 组件，让手柄向前发射射线，用于检测前方物体和 UI 元素。

### 卡片三：VR UI 适配

**标题**：第三步 —— 配置 TrackedDeviceGraphicRaycaster 与 XRUIInputModule

**正文**：
在需要交互的 Canvas 上挂载 TrackedDeviceGraphicRaycaster 组件，替换默认的 GraphicRaycaster，让它接收来自 XR 设备的射线事件。在场景的 EventSystem 上挂载 XRUIInputModule 组件，替换默认的 StandaloneInputModule，负责将射线事件分发给 UI 按钮。配置完成后所有 Button、Slider 等组件自动响应手柄射线的悬停和点击。

### Presenter 讲解节奏

1. "先创建 XR Origin，Building Blocks 一键生成，不用手搭"（XR 场景）
2. "左右手控制器上挂 XRRayInteractor，手柄就能发射射线点 UI"（射线交互）
3. "Canvas 的 GraphicRaycaster 换成 TrackedDeviceGraphicRaycaster，EventSystem 换成 XRUIInputModule——换两个组件 UI 就能接手柄事件了"（UI 适配）

### 一句话总结

> "创建 XR Origin、挂射线、换 UI 组件——三步搭完 VR 交互场景。"

### 配图建议

三步流程：①XR Origin 结构示意（左右手控制器+摄像机） → ②XRRayInteractor 射线示意 → ③Canvas 组件列表对比（GraphicRaycaster→TrackedDeviceGraphicRaycaster、StandaloneInputModule→XRUIInputModule）。

**正文**：

### 卡片三：近身搏斗

**标题**：03 / 面对死士

**正文**：

步兵从多个出兵点同时向城墙推进，每一路方阵由一名指挥官带队，小兵围绕指挥官保持阵型。阵型维护采用三个策略：以指挥官到城墙的方向为绝对基准轴计算每个小兵的位置，防止转弯时方阵被甩成弧线；小兵当前目的地与新位置偏离小于 1 米时不重新寻路，避免每帧高频计算；掉队超过 3 米的小兵临时加速到 1.3 倍速度快速归位。方阵抵达城墙后开始架设云梯，梯子在 2 秒内从平放旋转竖起，为后续死士翻墙提供路径。全场步兵上限 30 人，达到上限后不再生成。

**标题**：骑兵（EnemyHorse）—— 侧翼冲击城门的机动力量
