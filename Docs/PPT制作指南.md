## 第 3 页：行业之困

**标题**：行业之困：VR 体验与文旅叙事的双重瓶颈

**左栏 — VR 交互设计的两大极端**：
- ✗ **被动观看**：传统"360° 全景视频"让体验者沦为被动的摄像机，缺乏交互性
- ✗ **粗暴移植**：直接将 PC 游戏的"UI 点击"与"摇杆移动"移植到 VR，破坏沉浸感

**右栏 — 文旅与历史叙事的三大困境**：
- **遗址"沉默"**：静态遗址难讲述背后的英雄史诗
- **形式单一**：传统展板和讲解词冲击力有限
- **体验同质化**：缺乏互动性，难吸引年轻游客

**底部结论**：
> 市场急需一种"高沉浸、强交互、低眩晕、有内涵"的新型体验模式

## 第 7 页：战斗流程概览

**标题**：战斗流程：进入场景后自动展开的攻防战

**核心描述**：
玩家戴上头显进入战斗场景后，无需任何加载操作——BattleDirector 在 3 秒延时后自动启动整场战斗。战斗并非单一波次对抗，而是**四条线程同时运转**的复合攻防体系：步兵从正面持续压境、骑兵从侧翼冲击城门、箭雨无差别覆盖城头、精英死士伺机翻墙突破。玩家站在城墙上，面对的是从地面到空中的立体压力网。

**战斗时序（主讲人口述节奏）**：

```
进入 Scene2_city_v2
    │
    ▼ 3 秒后 —— 给玩家环顾环境、拿起弩的时间
BattleDirector.StartBattle()
    │
    ├─ 波次生成：多路步兵方阵从出兵点向城墙推进
    │   开场即启动，持续施压，上限 30 人同时在场
    │
    ├─ 箭雨循环：每 15±3 秒 200 支箭从天空射出
    │   压迫感的主要来源，逼玩家不能站桩
    │
    ├─ 骑兵冲锋：开局 10 匹从侧翼冲城门，之后每 8 秒补充
    │   分散注意力，城门被冲撞时有震动反馈
    │
    └─ 精英死士：隐藏跟随指挥官，梯子竖起后激活
    │   阶段性的"死亡倒计时"，制造剧情高潮
```

**玩家在城墙上能做的三件事（向评委强调：这在 VR 中是真实体能操作，不是按按钮）**：

| 行动 | 操作 | 效果 |
|------|------|------|
| 🏹 **射击压制** | 大黄弩——双手握持——扣扳机 | 命中 → 全局阻尼 0.1 → 全军停滞约 2s |
| 🛡️ **躲避箭雨** | 真实下蹲/侧移/缩头（真身体动作） | 被击中则闪红 + 僵直 1s |
| ⚔️ **面对死士** | （被追逐，无反击手段） | 被命中 → 丢武器 → 跳转结算 |

## 第 8 页：玩家交互详解

**标题**：玩家交互：大黄弩射击与受伤反馈

**核心描述**：
本页聚焦玩家在 VR 中的两种核心体感——"攻击"与"被攻击"。与 PC 游戏不同，这里的射击依赖真实的手部瞄准和扣扳机动作，受伤也通过视觉和操作反馈而不是血条数字来传达。**一切交互都基于身体动作，不依赖屏幕 UI。**

---

**大黄弩射击（CrossbowShoot）—— 一箭定全局**

**描述**：玩家通过 VR 手柄扣动扳机触发射击。箭矢从弩口生成后以 2000 力向前飞出。关键在于命中后的效果——它不是简单的"敌人减血"，而是触发**全局压制机制**：命中瞬间将 `globalDamping` 设为 0.1，全场敌人几乎静止，给玩家约 2 秒的喘息窗口。**一发命中 = 全体停滞**，这是战斗系统的核心正反馈。

```
VR 手柄扣动扳机
    │
    ▼
生成箭矢(firePoint 位置)
    │
    ▼
物理力发射 (AddForce 2000)
    │
    ├─ 命中敌人 → EnemyHealth.Die()
    │     └─ SiegeManager.TriggerSuppressingFire()
    │           └─ globalDamping = 0.1 → 全场停顿 ≈ 2s
    │
    └─ 未命中 / 飞出世界 → 5 秒后自动销毁（对象池回收）
```

**Presenter 一句话总结**："射中一箭，对面全停。射得越准，活得越久。"

---

**玩家受伤系统（PlayerHealth）—— 不靠血条，靠体感**

**描述**：VR 游戏中显示血条会破坏沉浸感，所以项目采用**纯体感反馈**来传递受伤信息：
- **中箭**时整个视野变红（一个透明球体包裹摄像机做透明度渐变），同时双手被"锁住"1 秒（禁用扳机和摇杆输入），模拟被箭射中时的震惊和僵直。5 秒后箭矢自动消失，玩家恢复行动。
- **被死士攻击**是更严重的惩罚——武器被强制脱手，1.5 秒后直接跳转结算场景，没有"残血苟活"的机会。这营造了死士的压迫感：被碰到就是结局。

| 受伤来源 | 体感表现 | 操作影响 | 恢复时机 |
|---------|---------|---------|---------|
| 🏹 **中箭** | 屏幕闪红（球体 Mesh 包裹摄像机，60%→0 透明度渐变） | 僵直 1s（双手扳机 + 摇杆被禁用） | 5s 后箭矢自动销毁 |
| 💀 **被死士攻击** | 闪红 + 武器脱手 | 丢武器后无法反击 | 1.5s 后跳转结算场景 |

**配图**：大黄弩瞄准射击第一人称截图 + 中箭闪红效果截图

**讲解提示**：讲这页时建议现场演示举弩瞄准的动作（不用真戴头显），让评委直观感受"VR 射击不是对准星而是真瞄准"。受伤部分强调"没有血条"这个设计决定。

## 第 9 页：战斗的多线程攻防

**标题**：战斗系统：四条线程同步推进

**四条战斗线程**：

**线程一：步兵方阵（SquadCommander）**
- 多路同时从出兵点推进
- 阵型维护：绝对基准轴 + 防抖 + 追赶
- 抵达城墙 → 架设云梯（2s Lerp 旋转竖起）
- 上限 30 人同时在场

**线程二：骑兵冲锋（EnemyHorse）**
- 从侧翼冲向城门，到城门消失
- 开局 10 匹冲锋，之后每 8 秒补 3 匹
- 上限 15 匹
- 受全局阻尼影响速度

**线程三：箭雨覆盖（ArrowRainVolley）**
- 每 15±3 秒一波，每波 200 支
- 抛物线弹道，飞行时间 1.8s
- 250 对象池复用
- 属于战场机制，非独立敌人类型

**线程四：精英死士（EnemyAssault）**
- 7 状态 FSM：跟随 → 跑向梯子 → 攀爬 → 翻墙 → 追逐 → 攻击
- 仅触发一次，命中即游戏结束
- 水平距离判定（忽略 Y 轴）

## 第 10 页：全局阻尼 — 战斗核心循环

**标题**：核心玩法：全局阻尼 — 压制与反压制

**核心描述**：
这是整款游戏**最核心的玩法闭环**。传统 FPS 游戏中，击杀敌人是一个局部事件——你打死了一个敌人，仅此而已。但在《孤城》中，每一次击杀都会对整个战场产生影响。这个机制叫做**"全局阻尼"**：它是一个 0.1 ~ 1.0 之间浮动的全局系数，直接控制所有敌方单位的推进速度。

设计意图是：**将玩家的射击精度转化为生存时间的可控变量**——你射得越准，战场压力就越低；射不准或不敢射，敌人就越逼越近。这不是一个"隐藏机制"，而是玩家能直观感受到的因果关系：射中一个敌人→敌军全部慢下来→压力骤减→阻尼恢复→压力回升→必须再次射中。

**核心循环图（建议放在页面正中央，大图）**：

```
     玩家射中敌军
         │
         ▼
  globalDamping = 0.1  ← 瞬间触发
         │
         ▼
  全军速度 × 0.1，几乎停滞  ← 深呼吸窗口
         │
         ▼
  阻尼以 0.5/s 缓慢恢复    ← 大约 2 秒有效窗口
         │
         ▼
  压力逐渐回升             ← 需要再次射杀
         │
         └─────────────→ 回到起点
```

**代码逻辑（展示用，缩小放在循环图下方）**：

```csharp
public void TriggerSuppressingFire()
{
    globalDamping = 0.1f; // 一箭射出，全场冻结
}

void Update()
{
    // 所有推进速度 = 基础速度 × 阻尼系数
    siegeProgress += baseAdvanceSpeed * globalDamping * Time.deltaTime;
    // 阻尼随时间恢复（约 2 秒回到正常）
    if (globalDamping < 1f)
        globalDamping += dampingRecoveryRate * Time.deltaTime;
}
```

**被阻尼影响的系统（逐一枚举，向评委展示覆盖面）**：

| 系统 | 公式 | 效果 |
|------|------|------|
| `SquadCommander` 步兵方阵 | `agent.speed = baseSpeed × damping` | 步兵停滞，方阵停止推进 |
| `EnemyHorse` 骑兵冲锋 | `chargeSpeed = baseSpeed × damping` | 骑兵减速，城门压力降低 |
| `攻城进度` 梯子架设 | `progress += advanceRate × damping × Δt` | 梯子竖起速度变慢 |

**战术意义**（Presenter 关键句）：
> "这不是一个射击游戏——这是一个**节奏管理游戏**。你的射击精度直接决定你的生存时间。射得准 = 喘口气，射不准 = 被淹没。"

**讲解提示**：本页是答辩的"高潮页"。建议先展示循环图，用手顺着箭头走一圈；然后强调"一行代码（`globalDamping = 0.1f`）控制了三个大系统"——这是四两拨千斤的设计，评委很吃这种简洁优雅的架构。

## 第 11 页：技术框架总览

**标题**：技术框架：轻量化高性能移动端方案

**核心描述**：
项目技术选型围绕一个核心约束展开：**在移动 VR 的有限算力上，跑出稳定 60fps 的水墨风格战场**。这决定了整个技术架构的走向——不追求画面写实，而是通过简洁的分层架构、事件驱动模块解耦、轻量手写 Shader 和对象池/FFR 等性能手段，在不牺牲品质的前提下压榨移动端性能。本页用一张四层架构图展示技术全貌，**图是主角，文字是辅助标注**，Presenter 需要能沿着层级讲清楚每一层的定位。

**口述路线**（跟着图从下往上讲一遍）：
1. "先从最底层看——Unity 2022 + PICO 3 移动 VR 是我们选定的硬件基础，URP 管线为移动端做了批量渲染优化"（指底部平台层）
2. "第二层是性能优化——对象池、FFR、按需禁用 NavMesh，这些在后面第 17 页会展开"（指性能层）
3. "第三层是业务逻辑——事件驱动 + 分层架构，BattleDirector 指挥四个子系统，模块间通过 Event Bus 解耦"（指业务逻辑层）

## 第 13 页：撒豆成兵 — 战场生成体系

**标题**：核心玩法：撒豆成兵 — NavMesh 校验的士兵生成体系

**核心描述**：
战场上同时存在 30 个步兵、15 个骑兵，它们必须在高低不平的野外地形上找到可站立的位置，不能重叠、不能卡在石头里、不能掉进 NavMesh 空洞。实现方案借鉴了"撒豆成兵"的思路——在出兵点周围随机撒点，用 NavMesh 射线做地面校验，只有命中 NavMesh 的"豆子"才落地成兵。

**两步校验流程**：

```
① Random.insideUnitCircle × radius → 二维随机点
         │
         ▼
   转换成三维世界坐标（XZ 平面随机，Y 取发射器高度）
         │
         ▼
② NavMesh.SamplePosition(randomPos, out hit, 5f, AllAreas)
         │
         ├─ ✅ 命中 → Instantiate 生成士兵 → 注册到 SquadCommander
         │
         └─ ❌ 未命中 → 丢弃（该点不可行走）
```

**关键代码（缩小放置）**：

```csharp
Vector2 randomPoint = Random.insideUnitCircle * scatterRadius;
Vector3 randomPos = transform.position + new Vector3(randomPoint.x, 0, randomPoint.y);

NavMeshHit hit;
if (NavMesh.SamplePosition(randomPos, out hit, 5.0f, NavMesh.AllAreas))
{
    GameObject soldier = Instantiate(enemyPrefab, hit.position, Quaternion.identity);
    commander.squadMembers.Add(soldier.GetComponent<NavMeshAgent>());
}
```

**三种生成场景（PPT 建议用三列布局）**：

| 场景 | 生成方式 | 特点 |
|------|---------|------|
| **步兵开局** | ArmyBeanScatterer 在出兵点撒豆 | 多路同时生成，上限 30 人 |
| **骑兵持续补充** | BattleDirector 协程，每 8 秒在随机出兵点补 3 匹 | 上限 15，达上限跳过 |
| **精英死士** | 预制体直接放置在场景中，由 BattleDirector 激活 | 唯一实例，不参与随机生成 |

**关键设计（每点都是一个可以展开讲的"技术亮点"）**：

| 设计点 | 为什么重要 | 实现方式 |
|--------|-----------|---------|
| **NavMesh 地面校验** | 野外地形起伏不平，直接随机生成会导致士兵卡在模型里或掉进 NavMesh 空洞 | `NavMesh.SamplePosition(randomPos, out hit, 5f, AllAreas)` — 只有命中 NavMesh 的"豆子"才落地成兵 |
| **撒豆随机性** | 每次开局敌人分布不同，增加战斗的不可预测感和重玩性 | `Random.insideUnitCircle × scatterRadius` 二维随机偏移，每次生成位置不同 |
| **三层差异生成策略** | 不同兵种对定位精度的要求不同——步兵需阵型整齐、骑兵需冲锋路线通畅、死士需精确触发位置 | 步兵撒豆（大规模随机）、骑兵定时补兵（随机出兵点）、死士预制体（精确放置，不参与随机） |
| **上限控制容错** | 防止场上实体数量超过移动端性能承载上限 | 步兵上限 30、骑兵上限 15，达上限跳过生成，保证 NavMeshAgent 活跃量可控 |

**Presenter 讲解节奏**：
1. "战场上 30 个步兵、15 个骑兵同时存在，它们怎么找到各自的位置？不是手工摆放——而是随机撒豆子"（引题）
2. "每个出兵点周围随机撒点 → NavMesh 探针校验地面 → 通过的才生成。这套流程保证每次开局敌人的分布都不一样"（解释机制）
3. "注意三种生成策略的差异——步兵大规模撒豆追求数量、骑兵按需补兵追求持续、死士精确放置追求可控。每种兵种用最适合自己的方式"（区分层次）
4. "NavMesh 校验这个技术细节值得评委注意：没有它，随机生成在野外地形上会产生大量运行时 bug——卡石头、掉空洞。一行 `NavMesh.SamplePosition` 解决了一类问题"（技术点睛）

**Presenter 一句话总结**：
> "不是在场景里摆 30 个预制体，而是在每个出兵点随机撒豆子——NavMesh 探针告诉你能站还是不能站。每次开局敌人的位置都不一样。"

**配图建议**：截图展示撒豆子的圆盘 Gizmo + NavMesh 蓝色地面 + 生成的士兵队列。也可以加一张对比图——没有 NavMesh 校验时士兵卡在石头里的 bug 截图（制造"笑点"）

## 第 14 页：精英死士 FSM

**标题**：核心玩法：精英死士 — 7 状态有限状态机

**核心描述**：
全局阻尼制造的是"量变"的压力——敌人多、走得快，但都在可控范围内。精英死士则负责制造"质变"的危机——它是唯一的、必杀的、无法通过"射得准"来化解的威胁。

死士的设计定位是**"阶段性的死亡倒计时"**：它一开始隐藏在普通步兵中跟随推进，不暴露任何特殊标识。当步兵搭好云梯后，死士被激活，开始执行一套从地面到城墙再到玩家面前的完整入侵路径。从它被激活的那一刻起，玩家就进入了一个"倒计时"状态——死士没有血条、不能被击杀，只有两种结局：它追到你，游戏结束；或者你在它追到你之前已经……输给其他敌人。

**7 状态流转（全页展示，大图）**：

```
Idle（隐藏状态）
  │ 玩家看不到它，混在步兵中跟随
  │ ActivateAndFollow(commander) ← 梯子竖起触发
  ▼
MarchingToWall（跟随指挥官推进）
  │ 仍然混在人群中，与普通步兵行为无异
  │ StartAssaultWithTarget(anchor) ← 到达城墙根
  ▼
RunningToLadder（暴露！跑向梯底）
  │ 突然加速脱离人群，玩家这时才意识到"有个不一样的"
  │ OnTriggerEnter(BottomTrigger)
  ▼
Climbing（攀爬云梯）
  │ 沿梯子向上，高度攀升，视野中逐渐变大
  │ OnTriggerEnter(TopTrigger)
  ▼
ClimbingEnd（翻墙 1.5s）
  │ 翻过城墙边缘，重新启用 NavMeshAgent
  │ agent.enabled = true
  ▼
ChasingPlayer（城墙上追逐）
  │ 水平距离判定（忽略 Y 轴，防 VR 身高差异）
  │ 距离 ≤ 1.5m
  ▼
Attacking（攻击）→ HitPlayer() → 结算场景
```

**关键设计（每点都是一个可以展开讲的"亮点"）**：

| 设计点 | 为什么重要 | 实现方式 |
|--------|-----------|---------|
| **水平距离判定** | VR 玩家身高不同，用 Y 轴判定死士"伸手够到没"会出错 | 所有距离检测忽略 Y 轴，只看 XZ 平面 |
| **Animation Event** | 攻击动画与伤害判定精确同步，不靠每帧检测 | `HitPlayer()` 在武器挥出帧由动画事件调用 |
| **武器拖尾墨迹** | 让攻击动作在黑白红水墨风格中视觉清晰可辨 | 攻击时开启 Trail Renderer，收刀关闭 |
| **保底触发** | 防碰撞体漏检的容错——距梯子够近但 trigger 没触发时强制执行 | `if (distToLadder < threshold) ForceClimb()` |

**Presenter 讲解节奏**：
1. "前几页讲的是'量'的压力——人多、箭密。但只有量不够，需要质的威胁。"（铺垫）
2. "所以我们设计了一个精英死士，它不能被击杀，被碰到就是结局。"（定性）
3. "它从隐藏 → 暴露 → 爬梯 → 翻墙 → 追你，整个过程大约需要 10 秒——这是给玩家的倒计时。"（制造紧张感）
4. "注意这里的关键：死士追你的时候，其他三条战斗线程还在同时运转。所以玩家面临的是'一边要射箭压制步兵，一边要躲箭雨，一边还要回头确认死士到哪了'的多任务压力。"（点出多线程协同的效果）

## 第 15 页：箭雨物理系统

**标题**：核心玩法：铺天箭雨 — 抛物线弹道与对象池

**箭雨生成流程**：

每一支箭的弹道由三段计算确定。先在 skyCloudBox 范围内随机取一个 XY 坐标，向下发射射线检测 wallLayer，命中点即为箭矢的落点。然后在 sourceFiringArea 的 BoxCollider 体积内随机采样一个点作为发射起点。知道起点和落点的坐标后，固定飞行时间为 1.8 秒，反向解算出初速度：水平分量等于水平距离除以飞行时间，垂直分量由垂直落差和重力加速度共同决定。解算完成后从对象池取出箭矢，设置速度向量并激活，同时挂载破空音效跟随箭矢飞行。

**弹道公式**：

```
velocityXZ = horizontalDistance / time
velocityY = (verticalDistance - 0.5 × g × t²) / t
```

**对象池管理**：

```
Awake() → 预生成 poolSize(250) 支箭 → Queue 管理
  ├─ GetArrowFromPool() → 取出/扩容
  ├─ ReturnArrowToPool() → 回收
  └─ 每 4~7 秒自动回池
```

**性能数据**：250 对象池避免 GC 抖动，波形 200 支箭/12~18s

## 第 16 页：Shader 与水墨视觉

**标题**：视觉风格：黑白红水墨 — Shader 技术详解

**三行布局**：

**水墨透明（InkCodeShader）**
```hlsl
half alpha = 1.0 - texColor.r; // 白→透明，黑→不透明
return half4(_InkColor.rgb, alpha * _InkColor.a);
```
- 白底黑墨贴图，红色通道反相为透明度
- `Blend SrcAlpha OneMinusSrcAlpha` 透明混合

**法线描边（SG_PropOutline）**
- 法线向量检测边缘 → Step 硬边裁切 → Lerp 混合
- 角色、可交互物体带"手绘感"轮廓

**旗帜飘动（FlagWind / InkFlagWind）**
```hlsl
// 顶点动画：sin(时间 × 风速 + 位置 × 频率) × 振幅
// 世界坐标错帧：不同位置旗帜异步飘动
float phase = worldPos.x * _Frequency
            + worldPos.y * _Frequency * _Stagger;
```

**材质分类**：
| 水墨透明类 | 描边类 | 旗帜类 |
| InkCodeShader | SG_PropOutline | FlagWind |
| 城墙、房屋 | 敌人、马、弩 | 水墨旗、布旗 |

## 第 17 页：性能优化

**标题**：性能优化：移动端 60fps 保障策略

**四大优化方向**：

| 方向 | 具体措施 | 效果 |
|------|---------|------|
| **对象池** | 箭雨预生成 250 支箭矢，每次发射从队列取出，命中或超时自动回收复用，全程无新对象创建。骑兵采用容量上限池开局生成 100 匹后由协程定时补充，到达城门或死亡后销毁，释放的容量由下一批填充。避免运行时频繁 Instantiate/Destroy 产生的 GC 抖动 | 大幅降低内存分配频率与性能开销 |
